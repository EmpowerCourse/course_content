1:00 PM Introduction to Source Control
  • Why?
  • git
    • Architecture
    • Install Client
    • Create Repository Host Account at github, bitbucket, beanstalk or another
    • Setup SSH access to eliminate the need to provide a username and password over https with every usage
      • open git bash from Windows Explorer (path doesn't matter right now)
      • ssh-keygen -t rsa -C "unique-id"
      • open the generated .pub file and copy entire contents
      • go to your repository host account and Add SSH Key
      • back in your client git bash session add the server to known hosts: ssh -T serverSiteName and answer 'yes' to the prompt
    • config your git profile so all of your interactions will be tagged appropriately in the repositories you interact with
      • git config --global user.name "Firstname Lastname"
      • git config --global user.email "your_email@youremail.com"
    • setup your conflict resolution tool (mergetool) via config
      • Download P4Merge (or another to your liking)
      • exec the following in a git bash: git config --global merge.tool p4merge
      • exec the following in a git bash: git config --global mergetool.p4merge.cmd 'p4merge.exe \"$BASE\" \"$LOCAL\" \"$REMOTE\" \"$MERGED\"'
    • Create an initial repository within your online git account (name is sacrosanct, think before you name)
    • clone to your local machine (either the HTTPS or SSH command will normally be copyable from your online repository view)
    • setup a .gitignore file appropriate to your repository
    • add/change/delete file(s) using an editor of your choice
    • add changes to snapshot your work
    • commit to create a point in time change set and identify it with a comment
    • pull to ensure you have the latest from the remote; if any changes were pulled in validate your changes are still valid/functional
    • push your changes to the remote
    • branch branchName
    • checkout branchName
    • compare to git checkout -b branchName
    • commit changes in a branch
    • merge branchToMergeIntoCurrentBranch
    • conflict resolution
      • mergetool (when necessary)
      • be sure to commit following a conflict merge
    • stashing
      • stash
      • stash list
      • stash apply (to apply to current state)
      • stash drop stashName (to delete)
      • stash pop (to apply and remove in one statement)
    • delete a branch: git branch -d branchName (use a capital D to ignore any unmerged changes)
    • See this SO post for options on reverting...
  • Subversion (svn)
    • Architecture
    • Install Client
    • Setup an online account at ProjectLocker or another provider of your choosing.
    • Create a svn project (name is just as important here)
    • Setup a local project directory
      • Create a directory in a desired project root on your local drive
      • Right click and choose SVN Checkout, specifying the repo URL and providing your credentials
      • Setup the /trunk, /branches and /tags subdirectories
      • add/change/delete file(s) in trunk using an editor of your choice
    • "Add" the new items
    • SVN Update and SVN Commit are somewhat similar to pull and push
    • Notice how stright deletions without an SVN delete will not be respected (perform a SVN delete)...
    • built-in conflict resolution is decent
    • Setup ignores: right click on a SVN directory and /Tortoise SVN/Settings and modify the global ignore string
    • Right click on trunk and choose Branch/Tag, specifying the /branches/branchName in the URL; Watch the Switch checkbox at the bottom, left!
    • Context-specific log details and the ability to easily deal with 1 file or 1 folder
    • Undo at the file level by "Update to Revision" from the log browser
    • Always make sure all pending changes are committed to the current branch before switching
    • To merge, switch to trunk and choose to Merge (reintegrate branch)
    • The merge will not commit to trunk, just perform the changes so you'll need to commit
    • Learn More
